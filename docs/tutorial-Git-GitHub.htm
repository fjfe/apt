¿Qué es GIT?

GIT es un software de control de versiones.
Su propósito es llevar registro de los cambios en archivos de computadora y coordinar el trabajo que varias personas (o una sola) realizan sobre archivos compartidos (o personales).
Existe la posibilidad de trabajar de forma remota y una opción para ello es GitHub.

Flujo de trabajo de GIT


Tenemos nuestro directorio local (una carpeta en nuestro PC) con muchos archivos. Git nos irá registrando los cambios de archivos o códigos cuando nosotros le indiquemos, así podremos viajar en el tiempo retrocediendo cambios o restaurando versiones de código, ya sea en Local o de forma Remota (servidor externo).

¿Qué es GitHub?

GitHub es una plataforma de desarrollo colaborativo para alojar proyectos (en la nube) utilizando el sistema de control de versiones Git.
Además cuenta con una herramienta muy útil que es GitHub Pages donde podemos publicar nuestros proyectos estáticos (HTML, CSS y JS) gratis.

Configurar el usuario y email global (Por defecto se usará éste en commit) 
	git config --global user.name "Paco Fdez"

git log

Trucos

Mostrar los commit realizados en una línea 
git log --oneline
Mostrar los commit realizados en una línea más elegante 
git log --oneline --decorate --all --graph
Mostrar sólo los archivos modificados 
git status -s

Diferencias entre -- y -
--decorate hace referencia a una palabra
-s hace referencia al comando o a varios comandos, -sa serían dos comandos diferentes

Mostrar información de la rama maestra. Ambos comandos hacen lo mismo 
git status -s -b
	git status -sb

Crear alias globales

Los alias sirven para crear atajos de comandos.
Podemos guardar diferentes alias de forma global y quedarán guardados en la configuración de Git.

Guardar el alias "lg" que ejecutará todo lo que está entre comillas 
git config --global alias.lg "log --oneline --decorate --all --graph"
Editar el archivo config (que incluye los alias creados) 
git config --global -e
Se edita con Vim, que es el editor de código en línea de comandos (habitual en Linux).

Salir del modo edición "Vim"
Para salir del modo edición de la líneas de comando presionar :q
En caso de realizar algún cambio sin guardar escribir :qa
:q! también sirve para salir sin guardar.

Mostrar el archivo config (que incluye los alias creados) 
git config --global -l
	user.email=sutanito@serv.com
	user.name=Paco Fdez
	alias.lg=log --oneline --decorate --all --graph
Realizar de una vez el add . y el commit con mensaje 
git commit -am "Mensaje del commit"
Editar un commit, como por ej: el mensaje 
git commit --amend

Comandos del editor Vim
i comenzar a editar
esc salir del modo edición
wq guardar y salir
q! salir sin guardar cambios

Viajes a través de los commit (Viajes en el tiempo 😉)

Supongamos que tenemos los siguientes commit:
  • f82f457 (HEAD -> master) mas comandos agregados
  • f52f3da nuevos comandos en fundamentos.md
  • e4ab8af mi primer commit
Viajar al commit especificado 
git reset --mixed f52f3da
Viajar al commit especificado y eliminar los cambios futuros 
git reset --hard f52f3da
Mostrar todos los cambios incluso si borramos los commit 
git reflog
Viajar al commit especificado y restaurar los archivos 
git reset --hard f52f3da
Revertir los cambios en un archivo específico, aunque no hicimos un commit 
git checkout -- nombreArchivo.ext
Destruir todos los cambios sin haber realizado un commit 
git reset --hard

Renombrar archivos

Cambiar el nombre de un archivo 
git mv nombreOriginal.ext nombreNuevo.ext

Mover archivos

Mover un archivo a otra carpeta 
git mv nombreOriginal.ext carpetaDestino/nombreNuevo.ext

Eliminar archivos

Borrar un archivo 
git rm nombreArchivo.ext

Ignorar archivos y carpetas

Para no hacer seguimiento de algunas carpetas y/o archivos, debemos crear un archivo de texto llamado .gitignore con un contenido similar al siguiente. 
	arhivo.js        // Ignora este archivo
	*.js             // Ignora todos los arhivos con extensión .js
	notas/           // Ignora toda esta carpeta

Ramas o branch

Mientras no se creen ramas, sólo existe una llamada "master", que debería ser la rama 'consolidada' de la que pueden partir otras y a la que se unirán cuando sus cambios sean estables.


Crear una rama 
git branch nombreRama
Mostrar la rama activa 
git branch
Cambiar la rama activa 
git checkout nombreRama

Atajo
Podemos crear una nuevaRama y viajar a ella con un solo comando:
git checkout -b nuevaRama

Para unir la rama master con la nueva, tenemos que estar en la master y ejecutar el comando merge: ¿¿¿??? 
	git checkout master
	git merge nombreRama
Eliminar una rama 
git branch -d nombreRama

Tags

Con los tags podemos hacer versiones de nuestro proyecto.

Crear un tag 
git tag versionAlpha -m "versión alpha"
Mostrar los tags 
git tag
Borrar tag 
git tag -d nombreTag
Hacer una versión en un commit anterior ej: f52f3da 
git tag -a nombreTag f52f3da -m "version alpha"
Mostrar información del tag 
git show nombreTag

Políticas de ramas

Las dos formas más habituales de utilizar Git para controlar la evolución de versiones software son las siguientes.

Git flow

Se caracteriza porque el código fuente principal se encuentra en una rama llamada “Develop”, con un acceso bastante estricto, y los desarrolladores crean ramas, llamadas “Feature”, tomando como base la rama “Develop”.

Antes de hacer una operación merge de la rama “Feature” con la rama “Develop” se abre una discusión entre todos los desarrolladores para aceptar todos los cambios en la rama “Develop”.

El código fuente de la rama “Develop”, que va a ser definitivo de la versión sobre la que se trabaja, será separado en una rama llamada “Release”. El código final almacenado en la rama “Release” es probado y sus errores son corregidos sobre la propia rama “Release”. Al código fuente de la versión definitiva de la rama “Release” se le hace una operación de merge con la rama “Master”, que es la rama que contiene las versiones liberadas.

Este tipo de control de versiones se suele usar cuando:

  • El equipo de desarrollo está compuesto por muchos miembros junior.
  • El proyecto sobre el que se trabaja es Open Source.
  • Productos en producción.

Trunk based development

Se caracteriza porque todos los desarrolladores trabajan sobre una única rama de código, conocida como “Master”, y suben los cambios que realizan a dicha rama.

A las versiones finales de código fuente se les crea una rama específica para dicha versión. Todas las ramas de versiones son conocidas como ramas “Release”.

  • Se necesita desarrollo y liberación rápidos.
  • El equipo de desarrollo está compuesto por miembros senior.
